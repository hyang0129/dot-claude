# Research Subskill — Best-Practice Probe for an Area

Invoked from setup Phase 2 (per-area drafting), refinement (whole-area `[TBD]` resolution), or amendment (adding a new convention or area). Surfaces current best practices and alternatives the user may not have considered, so the drafter can present an informed batch instead of guessing from training data alone.

This subskill is **advisory**. Its output feeds drafting; it does not gate any decision. The user always has the final word.

---

## When to invoke (recap)

The calling subskill decides when to invoke. Typical triggers:

- Low confidence on best practice for the user's stack in this area.
- Unfamiliar or recent stack version where training data may be stale.
- The user asked "what are my options for X?" in-line.
- A niche or fast-moving area (e.g., LLM tooling, JS frameworks).

Skip when:
- The user has already stated a clear preference for the area.
- The area is well-trodden and stable (lockfile use, language version pinning, etc.).
- A confirmed convention candidate already covers the area.

---

## Inputs (passed by the calling subskill)

- `AREA` — the area being researched (e.g., "Web framework / API layer").
- `STACK` — the project's stack & context block from Phase 1, verbatim.
- `USER_PREFERENCES` — anything the user has stated in this session about this area (may be empty).
- `CONSTITUTION_LAWS` — the law list, in case a law constrains the answer.

---

## Procedure

Run as a single subagent invocation. The subagent uses available web search / web fetch tools if the area is fast-moving and training data may be stale; otherwise it can answer from training data alone.

The subagent's prompt:

```
You are advising on engineering conventions for a project. The author is about to
draft conventions for one area of their codebase, and wants a brief best-practice
survey before they pick.

Area: <AREA>
Project stack & context:
<STACK>

User preferences expressed so far for this area:
<USER_PREFERENCES — verbatim, or "(none)">

Constitutional laws (constraints the chosen approach must not violate):
<CONSTITUTION_LAWS>

Return a structured advisory with three short sections.

## Best practices for this stack and area

  Two to four current best-practice recommendations. Each: one-line rule + one-line
  rationale. Bias toward what is conventional in <STACK> as of today. If "best
  practice" is contested in this area (e.g., framework choice), say so and present
  the leading two or three options with the trade-off.

## Alternatives the user may not have considered

  Up to three options the author might be unaware of: newer entrants, stack-specific
  libraries, or trade-off-driven picks (e.g., "consider Litestar instead of Flask
  if you want async-first"). Each: name + one-line reason it might fit + one-line
  reason it might not.

  If you cannot name a credible alternative the author is likely unaware of, say
  so and skip the section. Padding this section with weak suggestions is worse than
  leaving it empty.

## Common gotchas in this area

  Two to four pitfalls the author should know about when implementing conventions
  in this area, regardless of which option they pick. One line each.

## Constraints check

  If any of the recommendations or alternatives appear to conflict with the listed
  constitutional laws, flag them: "Option X may conflict with Law N because <reason>."
  This is advisory — the calling subskill will run the formal conflict pass at emit
  time. If nothing flags, write "No apparent conflicts."

Constraints on your output:

- Do not invent libraries, tools, or frameworks. If you do not know whether something
  exists or is current, say so.
- Do not recommend on the basis of popularity alone. Tie each recommendation to a
  rationale grounded in the stack or the area's known trade-offs.
- Do not push the user toward a particular pick. Surface options; the user decides.
- Keep each section short. The output goes inline into a drafting batch — verbose
  surveys overwhelm the user.
- If the area is genuinely "use the obvious thing, no real alternatives" (e.g., "use
  a lockfile"), say so plainly. Do not manufacture a debate.
```

---

## Output handling

The calling subskill receives the advisory and:

1. Caches it to `CONVENTIONS.research.md` at the same directory as `CONVENTIONS_PATH`. Append, with a heading per area:

   ```markdown
   ## <AREA> — researched <YYYY-MM-DD>

   <full advisory output verbatim>
   ```

   If `CONVENTIONS.research.md` doesn't exist, create it with a brief header:

   ```markdown
   # Conventions Research Cache
   <!-- Generated by /refine-conventions research subskill. Reference material only. -->
   ```

2. Surfaces relevant pieces inline to the user as part of the drafting batch. Typical pattern: the advisory's "Best practices" become draft conventions; "Alternatives" become `Notes:` lines on the relevant draft entries; "Gotchas" can become draft conventions in their own right (e.g., "Don't store secrets in env files committed to the repo") or `Notes:` lines.

3. Does not block on the advisory. If the user wants to pick something not in the survey, that is fine — record it.

---

## Closing rules

- Do not run research for areas where the user already volunteered a preference. Wasted round-trip and risks confusing the user with options they already declined.
- Do not present the raw advisory output to the user. Translate it into draft conventions and surface what's useful inline. The cache file is the raw record.
- Do not let research delay drafting indefinitely. If the advisory subagent takes too long or returns thin output, proceed with the draft using best judgment and the cached output as a footnote.
